## Overview

AutoGen (Microsoft Research) is a framework for building multi-agent systems where multiple LLM agents collaborate on complex tasks through conversation. Its relationship with the filesystem is primarily about workspace coordination: when multiple agents need to share state, pass artifacts, and coordinate work, files are a natural shared medium.

## The AutoGen Workspace Model

AutoGen's core model is conversation-driven — agents exchange messages, and the "workspace" is implicitly the shared context those conversations create. However, for persistent state, file sharing, and handoffs between agents, AutoGen agents use the filesystem as a coordination layer.

A typical AutoGen multi-agent pipeline for software tasks:
1. **Orchestrator agent** receives a high-level task, writes a plan to `task_plan.md`
2. **Research agent** reads the plan, performs research, writes findings to `research_notes.md`
3. **Coding agent** reads the plan and research notes, writes code to the target files
4. **Review agent** reads the code, writes feedback to `review.md`
5. **Orchestrator** reads review, updates plan, loops or completes

Each agent reads from and writes to files as a shared workspace — the files are the coordination interface between agents that otherwise communicate only through messages.

## Code Execution Workdir

AutoGen's `ConversableAgent` supports a `code_execution_config` parameter that includes a `work_dir` — a directory where code artifacts, execution results, and intermediate outputs are written. This creates an implicit filesystem workspace for the agent session:

```python
from autogen import ConversableAgent

agent = ConversableAgent(
    name="code_agent",
    code_execution_config={
        "work_dir": "workspace/",
        "use_docker": False  # or True for sandboxing
    }
)
```

Code generated by the agent is written to `work_dir`. Execution results are captured. Other agents can read the outputs from the same directory.

## Memory in AutoGen

AutoGen (v0.4+) introduced a formal memory abstraction. The `Memory` protocol supports different backends:

- **In-memory** — conversation buffer, ephemeral
- **ChromaDB** — vector-based semantic memory, persistent
- **Custom** — any backend implementing the Memory interface

For file-based memory specifically, teams implement custom Memory backends that read from and write to Markdown files. This is not built-in — it's an integration point. AutoGen's philosophy is "bring your own memory backend" rather than prescribing one.

## The Manus Pattern (File-Based Planning)

The Manus agent (acquired for ~$2B) popularized a specific file-based planning pattern for long-running autonomous agents that influenced AutoGen-style frameworks:

```
workspace/
├── task_plan.md      # Goals, steps, progress checkmarks
├── notes.md          # Research findings, observations  
└── output.{ext}      # Deliverable (the actual output file)
```

The agent updates `task_plan.md` as it works — checking off completed steps, noting blockers, updating the plan as new information arrives. This gives the agent a "memory of what it was doing" that survives context window limits during long tasks: when the agent needs to continue a multi-step task, it reads `task_plan.md` rather than relying on the full conversation history.

## Multi-Agent File Coordination Patterns

For multi-agent AutoGen systems, file coordination requires discipline to avoid race conditions and conflicting writes:

**Sequential writes** — assign each agent a distinct output file; the next agent reads the previous agent's output. No shared write access, no conflicts.

**Timestamp-namespaced files** — `agent1_output_20260405_143022.md` — multiple agents can write without collision; orchestrator reads latest.

**File-based signaling** — agents write "ready" files to signal completion: `research_complete.flag`. Other agents poll for these signals before proceeding.

**Git-mediated coordination** — each agent commits its work to a branch; the orchestrator merges. Conflicts surface as merge conflicts and require orchestrator resolution. See [[wiki/agent-filesystems/patterns/gitops-for-agents|GitOps for Agents]].

## Key Takeaways

- AutoGen uses the filesystem primarily as a coordination layer in multi-agent systems — files are the handoff mechanism between agents
- The `work_dir` configuration creates a shared workspace for code execution and artifact exchange
- AutoGen's memory abstraction is backend-agnostic; file-based memory requires a custom implementation
- The Manus three-file pattern (task_plan + notes + output) is influential for long-running autonomous agent design
- Multi-agent file coordination requires explicit conventions to avoid race conditions — sequential writes or git-mediated coordination are the practical solutions
